# Node.js

## Overview
Node.js is a JavaScript runtime built on Chrome's V8 JavaScript engine. It allows developers to run JavaScript on the server side, enabling scalable network applications. Node.js uses an event-driven, non-blocking I/O model, making it lightweight and efficient for data-intensive real-time applications.

## Why It Matters
Node.js revolutionized backend development by allowing JavaScript to be used across the full stack. Its non-blocking I/O architecture enables high throughput and scalability, making it ideal for real-time applications like chat, gaming servers, APIs, and microservices. The vast npm ecosystem provides over a million packages, accelerating development.

## Real-world Usage
- **Web Servers and APIs**: Express.js, Koa, Hapi.js frameworks built on Node.js are used to create RESTful APIs and server-side rendering.
- **Real-time Applications**: Chat applications, live streaming, and online gaming leverage WebSocket libraries like Socket.io.
- **Microservices**: Node.js's lightweight nature makes it suitable for microservices architectures.
- **Tooling**: Build tools like Webpack, Gulp, and Grunt are powered by Node.js.
- **Serverless**: AWS Lambda, Azure Functions support Node.js for event-driven computing.

## Code Example
Here's a simple HTTP server using Node.js built-in http module:

```javascript
const http = require('http');

const server = http.createServer((req, res) => {
  res.writeHead(200, { 'Content-Type': 'text/plain' });
  res.end('Hello World\n');
});

server.listen(3000, '127.0.0.1', () => {
  console.log('Server running at http://127.0.0.1:3000/');
});
```

## Common Mistakes
- **Blocking the Event Loop**: Using synchronous operations (e.g., fs.readFileSync) in request handlers can delay all other requests.
- **Ignoring Error Handling**: Forgetting to handle errors in callbacks or promises can lead to unhandled exceptions and crashes.
- **Callback Hell**: Deeply nested callbacks make code hard to read and maintain; use Promises or async/await.
- **Memory Leaks**: Accidentally holding references to objects in global scope or closures can cause memory leaks over time.
- **Not Using Cluster Module**: For CPU-intensive tasks, not utilizing multi-core systems can limit performance; use the cluster module to fork processes.

## Interview Question
**Question**: Explain the difference between `process.nextTick()` and `setImmediate()` in Node.js.

**Answer**: 
- `process.nextTick()` schedules a callback to be invoked in the same phase of the event loop, after the current operation completes but before the event loop continues. It has higher priority than I/O events.
- `setImmediate()` schedules a callback to be executed in the next iteration of the event loop, after I/O events. It is used to defer execution until after the current poll phase.

Thus, `process.nextTick()` runs before any additional I/O events (including timers), while `setImmediate()` runs after I/O events in the next loop iteration.

## Key Takeaways
- Node.js enables JavaScript on the server with a non-blocking, event-driven architecture.
- It excels in I/O-heavy applications and real-time systems due to its asynchronous nature.
- The npm ecosystem is vast, providing reusable packages for almost any need.
- Avoid blocking the event loop; use asynchronous APIs and handle errors properly.
- For CPU-intensive tasks, consider using worker threads or the cluster pattern to utilize multiple cores.
